# Участие в проекте

Благодарим за интерес к проекту «Шаблон каталога курса для студентов»! Ваша помощь делает его лучше.

## Как я могу внести вклад?

Есть несколько способов помочь проекту:

1.  **Сообщение об ошибках.** Если вы нашли ошибку, создайте **Issue** на GitHub с подробным описанием проблемы и шагами для её воспроизведения.
2.  **Предложение улучшений.** Если у вас есть идея, как улучшить шаблон, создайте **Issue** с меткой `enhancement`.
3.  **Исправление ошибок и добавление нового функционала.** Мы приветствуем Pull Request (PR).

## Процесс разработки

Этот проект придерживается строгих стандартов для поддержания качества и согласованности кода.

### 1. Стандарты и инструменты

*   **Система контроля версий:** [Git Flow](README.md#используемые-стандарты-и-программные-продукты).
*   **Версионирование:** [Семантическое версионирование](https://semver.org/lang/ru/).
*   **Коммиты:** [Общепринятые коммиты (Conventional Commits)](https://www.conventionalcommits.org/ru/v1.0.0/). Для упрощения используйте `git-cz` (см. [инструкцию по установке](README.md#общепринятые-коммиты)).
*   **Ветки:** Основные ветки `main` и `develop`. Новые функции и исправления создаются в ветках `feature/`, `bugfix/` или `hotfix/` согласно Git Flow.

### 2. Начало работы (для внешних контрибьюторов)

1.  Сделайте **Fork** репозитория.
2.  Клонируйте свой форк на локальную машину:
    ```bash
    git clone --recursive git@github.com:<ваш-username>/course-directory-student-template.git
    ```
3.  Добавьте исходный репозиторий как удалённый (upstream):
    ```bash
    git remote add upstream git@github.com:yamadharma/course-directory-student-template.git
    ```
4.  Убедитесь, что у вас установлены все необходимые инструменты, описанные в [README.md](README.md#дополнительное-программное-обеспечение) (Quarto, Node.js, commitizen, standard-version и др.).

### 3. Работа над изменениями (для участников с прямым доступом)

1.  Переключитесь на ветку `develop` и обновите её:
    ```bash
    git checkout develop
    git pull origin develop
    ```
2.  Начните новую фичу или исправление с помощью Git Flow (или создайте ветку вручную):
    ```bash
    git flow feature start моя-новая-функция
    # или для исправления:
    git flow bugfix start исправление-ошибки
    ```
3.  Вносите изменения, делая коммиты с помощью `git-cz`:
    ```bash
    git add .
    git-cz
    ```
    Следуйте инструкциям для создания правильно оформленного сообщения коммита.
4.  Периодически синхронизируйте свою ветку с `develop`, чтобы избежать конфликтов.
5.  Когда работа завершена, завершите фичу через Git Flow:
    ```bash
    git flow feature finish моя-новая-функция
    ```
    Это автоматически сольёт вашу ветку в `develop`.
6.  Отправьте изменения в свой репозиторий:
    ```bash
    git push origin develop
    ```

### 4. Создание Pull Request

1.  Убедитесь, что ваша ветка (например, `feature/...`) полностью готова и протестирована.
2.  Создайте Pull Request из вашей ветки (в вашем форке) в ветку `develop` исходного репозитория.
3.  В описании PR подробно укажите, какие изменения вы сделали и почему.
4.  Дождитесь рецензии. Будьте готовы ответить на комментарии и, возможно, внести дополнительные правки.

### 5. Релизы

Созданием релизов (обновлением версии, `CHANGELOG.md` и тэгов) занимаются поддерживающие проект. Это делается с помощью `standard-version` и Git Flow, как описано в [README.md](README.md#использование-git-flow).
